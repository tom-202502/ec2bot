# Make Plan — 分阶段实施规划

你是一个 **编排器（Orchestrator）**。创建 LLM 友好的分阶段计划，使其可以在新的上下文中被逐阶段执行。

## 何时使用
- 即将开始一个复杂功能的开发
- 任务涉及多个文件、多个步骤
- 需要在动手前先理清思路和依赖关系
- 搭配 `/do` 使用：先 `/make-plan` 规划，再 `/do` 执行

## 委派模型

用子代理做 **事实收集**（文档、示例、函数签名、grep 搜索）。
编排器负责 **综合与计划编写**（阶段划分、任务描述、最终措辞）。

### 子代理报告契约（强制）

每个子代理的返回必须包含：
1. **查阅的来源**（文件路径 / URL）以及读取了什么
2. **具体发现**（精确的 API 名称 / 函数签名；精确的文件路径 / 位置）
3. **可复制的代码片段位置**（示例文件 / 具体行号）
4. **置信度说明** + 已知盲区（可能遗漏了什么）

如果子代理只给结论不给来源 → 拒绝并重新部署。

## 计划结构

### Phase 0：文档发现（必须第一个执行）

在规划实施之前，部署"文档发现"子代理：
1. 搜索并阅读相关文档、示例和已有模式
2. 识别实际可用的 API、方法和签名（不是猜测的）
3. 输出"允许的 API 列表"，引用具体的文档来源
4. 标注反模式：哪些方法 **不存在**、哪些参数 **已废弃**

编排器将所有发现汇总为 Phase 0 输出。

### 每个实施阶段必须包含

1. **要实现什么** — 任务描述应导向"从文档复制模式"，而非"改造现有代码"
   - 好的：`从 docs/examples.ts:45-60 复制 V2 session 模式`
   - 差的：`把现有代码迁移到 V2`
2. **文档引用** — 引用具体文件和行号作为参考模式
3. **验证清单** — 如何证明这个阶段完成了（测试、grep 检查）
4. **反模式防护** — 这个阶段 **不要** 做什么（虚构的 API、未文档化的参数）

### 最终阶段：全量验证

1. 验证所有实现是否匹配文档
2. grep 检查已知的反模式
3. 运行测试确认功能正常

## 核心原则

| 原则 | 说明 |
|------|------|
| 文档可用 ≠ 已使用 | 明确要求阅读文档，不要假设子代理已经读过 |
| 任务描述决定质量 | 把代理引向文档，而不只是结果 |
| 验证 > 假设 | 要求证据，不要假设 API 存在 |
| 阶段自包含 | 每个阶段独立携带自己的文档引用 |

## 反模式防护

- 不要发明"应该存在"的 API 方法
- 不要添加文档中没有的参数
- 不要跳过验证步骤
- 不要在不查看示例的情况下假设结构
